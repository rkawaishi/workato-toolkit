# Connector SDK overview

Official: https://docs.workato.com/en/developing-connectors/sdk.html

## What is a custom connector

A self-built connector for connecting to applications that Workato does not provide out of the box. It is written as a Ruby DSL and defines triggers, actions, and authentication.

## Development options

### 1. Web editor (Cloud SDK)
- Develop in the browser from Tools > Connector SDK in the Workato UI
- CodeMirror editor (syntax highlighting, error display)
- Interactive testing via the Test code tab

### 2. SDK CLI (local development)
- Install with `gem install workato-connector-sdk`
- Develop and test locally, then upload via `workato push`
- Unit tests with RSpec + VCR
- Git integration, CI/CD pipeline support

## SDK CLI commands

### SDK CLI (`gem install workato-connector-sdk`)

Used for local testing and development. Because the `workato` command name collides with the Platform CLI, run via `bundle exec`.

| Command | Description |
|---|---|
| `bundle exec workato new <PATH>` | Create a new connector project |
| `bundle exec workato exec <PATH>` | Run and test a connector lambda block |
| `bundle exec workato edit <PATH>` | Edit an encrypted file |
| `bundle exec workato generate <SUBCOMMAND>` | Generate code from templates |

### Uploading to / downloading from Workato (API helper recommended)

Use the API helper for push and pull. It authenticates via the Platform CLI profile, so gem-side authentication (API Client token) is not required.

```bash
# Push: create new
python3 scripts/workato-api.py sdk push --connector connectors/<name>/connector.rb --title "<Title>"

# Push: update an existing connector
python3 scripts/workato-api.py sdk push --connector connectors/<name>/connector.rb --connector-id <id>

# Pull: download a connector that exists in the workspace
python3 scripts/workato-api.py sdk pull --connector-id <id>
# (or by name)
python3 scripts/workato-api.py sdk pull --name <name>
```

> **Note**: `bundle exec workato push` / `workato sdk pull` are still usable, but they require a separate API Client token.

## Project structure

Files generated by `workato new <name>`:

```
<name>/
├── connector.rb          # Main connector definition
├── settings.yaml.enc     # Credentials (encrypted)
├── settings.yaml         # Credentials (plain text, for development)
├── master.key            # Encryption key (must be in .gitignore)
├── Gemfile               # Ruby dependencies
├── spec/                 # Test files
│   ├── spec_helper.rb    # RSpec + VCR config
│   ├── connector_spec.rb # Connector tests
│   └── cassettes/        # VCR recordings (HTTP mocks)
└── .rspec
```

## Prerequisites

- Ruby 2.7.x / 3.0.x / 3.1.x (for local testing)
- Pushing via the API helper only requires Platform CLI profile authentication (no Ruby needed)
- Using `bundle exec workato push` additionally requires an API client (with "Get details" permission)

## Environment setup pitfalls

### On Ruby 4.0, default gems are no longer bundled

To run `workato-connector-sdk` 1.3.x on Ruby 4.0 (which is in preview as of 2026-04), add the following to `Gemfile` explicitly. On Ruby 3.x these were bundled/default gems, so the issue typically surfaces during an upgrade as `LoadError: cannot load such file -- csv` and similar.

```ruby
# Gemfile
gem 'csv'
gem 'base64'
gem 'bigdecimal'
gem 'logger'
gem 'drb'
gem 'ostruct'
gem 'mutex_m'
```

### On macOS, `charlock_holmes` requires `icu4c@78` to build

If `bundle install` fails on `charlock_holmes`, it is an ICU version mismatch. Install the matching version with Homebrew and expose the PKG_CONFIG variables.

```bash
brew install icu4c@78

export PKG_CONFIG_PATH="$(brew --prefix icu4c@78)/lib/pkgconfig:$PKG_CONFIG_PATH"
export LDFLAGS="-L$(brew --prefix icu4c@78)/lib"
export CPPFLAGS="-I$(brew --prefix icu4c@78)/include"

bundle install
```

Versions offered by Homebrew change over time, so confirm the current list with `brew search icu4c` before installing.

### A `connector.rb` with Japanese comments needs `LANG/LC_ALL=UTF-8`

If the locale is left at C when running `bundle exec workato exec`, Japanese comments cause a parse error. Set UTF-8 explicitly in the run command, in `.envrc`, or in CI config.

```bash
LANG=en_US.UTF-8 LC_ALL=en_US.UTF-8 bundle exec workato exec test
```
