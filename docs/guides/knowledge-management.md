# Knowledge management guide

How to grow the toolkit's knowledge base and how to operate it.

## Why knowledge management matters

Workato recipe JSON has the following characteristics:

- It contains **many fields that are not documented in the official docs** (such as the concrete structure of `extended_output_schema`)
- Some **values can only be set from the UI** (such as `pick_list` choices or dynamic schemas)
- Each connector has **its own quirks** (field naming conventions, differences between required and optional, etc.)

These become clear as you develop, but tend to stay in individual memory. This toolkit accumulates that tacit knowledge as documentation so it can be applied to the next recipe generation.

No matter which editor (Claude Code / Cursor / Codex CLI / Gemini CLI) you learn in, knowledge is accumulated under `docs/` (kit canonical) and `org/docs/` (organization-specific corrections and additions). Files under `.claude/rules/` and the equivalent directories for each editor are pre-generated by the kit maintainer with `python3 scripts/sync_agents.py` and distributed to users via symlinks, so users do not need to run syncs themselves. To update the kit, run `git submodule update --remote kit && bash kit/setup.sh`.

```
│   ├── recipe-patterns/    # Recipe construction patterns (generic)

│   └── workspace-management.md  # Workspace management
└── connector-sdk/          # Connector SDK
    ├── overview.md
    └── connector-rb.md
```

In addition, organization-specific knowledge lives at:

```
connectors/docs/                            # Custom connector specs
org/docs/                                   # Overlay over kit canonical docs/
├── connectors/<provider>.md                #   Corrections/additions to kit version, org-specific info
├── logic/<topic>.md                        #
├── platform/<topic>.md                     #
├── patterns/recipe-patterns/<name>.md      #   Recipe construction patterns recorded by the org
├── patterns/deployment-guide.md            #
└── learned-patterns.md                     #   Temporary buffer on the org side
projects/docs/patterns/                     # Legacy (records from an older version, read-only)
```

For the lookup and write conventions of `org/docs/`, see `@.claude/rules/org-knowledge-overlay.md`.
**Users do not edit the kit canonical `docs/` directly** (that would mean changing the kit submodule).

## Running the learning cycle

### 1. `/learn-recipe` — learn from a recipe

Run this after pulling a recipe that was adjusted in the UI.

```
/pull-project                         # Bring UI changes down locally
/learn-recipe <project-name>   # Learn from the entire project
```

**What gets learned:**

| Finding | Destination | Example |
|---|---|---|
| Field definition | `org/docs/connectors/<provider>.md` | The Salesforce Account has a `BillingCity` field |
| New action | `org/docs/connectors/<provider>.md` | Jira has a `Create sprint` action |
| datapill pattern | `org/docs/logic/data-pills.md` | A new reference pattern `#{_('data.xxx.yyy')}` |
| JSON structure | `org/docs/learned-patterns.md` | A new way to use `toggle_cfg` |
| Deployment insight | `org/docs/patterns/deployment-guide.md` | Cases where push fails under specific conditions |

**Key points:**
- All write destinations live under `org/docs/` (do not touch the kit canonical `docs/`)
- Deduplicate: do not append information already known in the kit version or the org version
- `org/docs/learned-patterns.md` is a temporary staging area, a holding spot until items are routed to the right destination

### 2. `/learn-pattern` — record a construction pattern

Record an expert's "when this happens, build it this way" insight as a pattern.

```
/learn-pattern
```

It asks for the pattern's details interactively and documents them in the following structure:

```markdown
# Pattern name

## When to use it
- Condition 1
- Condition 2

## Recipe diagram
trigger -> step1 -> step2 -> ...

## Step configuration
Details of each step

## Design rationale
Why this configuration was chosen

## Pitfalls
Points to watch out for
```

**Where patterns are stored:** `org/docs/patterns/recipe-patterns/<name>.md` (both generic and org-domain patterns are consolidated here)

The distinction between generic and org-domain is expressed in the pattern body's "Scope" section (not by path).
Patterns that were recorded in older versions under `projects/docs/patterns/` are also read at load time (backward compatible).

**How patterns are used:**
- Referenced automatically during step design in `/create-recipe`
- Proposed as candidates during architecture decisions in `/plan`
- Patterns are composable blocks, not monolithic templates

### 3. `/sync-connectors` — sync connector information

Fetches the latest connector information and updates the documentation.

```
/sync-connectors salesforce    # Sync Salesforce only
/sync-connectors --custom      # Sync custom connectors only
/sync-connectors --all         # Sync all connectors
```

**Pre-built connectors:**
- Fetches trigger / action metadata from the Workato API
- Updates `docs/connectors/<provider>.md`
- Hand-written field detail sections are preserved

**Custom connectors:**
- Parses `connectors/*/connector.rb` via Ruby DSL parsing
- Generates and updates `connectors/docs/<name>.md`
- Extracts `input_fields`, `output_fields`, and `object_definitions`

## Recommended workflow

### When using a new connector for the first time

```
1. /sync-connectors <provider>     # Fetch connector information
2. /create-recipe                  # Generate the recipe
3. /push-project --start                  # Deploy
4. Adjust in the UI                # Pick from pick_list, fine-tune fields
5. /pull-project                          # Pull changes
6. /learn-recipe                   # Learn from the adjustments
```

From the second time on, steps 1, 4, 5, and 6 are unnecessary, and a correct recipe can be generated directly from the knowledge base.

### When you notice a recurring construction pattern

```
1. /learn-pattern                  # Record the pattern
2. From now on, /create-recipe uses it automatically
```

### When taking over a project

```
1. /pull-project --all                    # Pull every project
2. /learn-recipe <project>         # Learn from each project
3. /catalog                        # Inventory shared assets
```

## Keeping the knowledge high quality

- **Avoid duplication**: every skill checks existing content before appending
- **Place items in the right location**: `docs/learned-patterns.md` is a staging area; write discoveries directly into the proper document
- **Update stale information**: when a connector's spec changes, overwrite it with `/sync-connectors`
- **Distinguish organization-specific from generic**: consolidate patterns under `org/docs/patterns/recipe-patterns/`, and express generic vs. org-domain in the body's "Scope" section (used to decide whether to upstream into the kit)
