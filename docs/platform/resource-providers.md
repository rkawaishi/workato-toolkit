# Resource provider registry

Reference for automatically retrieving information about connected resources via external tools during recipe design (`/create-recipe`, `/design`).

## Overview

Some recipe input fields cannot be filled in unless the user knows the exact ID or field name (Jira project key, Slack channel ID, etc.). When an MCP server or CLI tool is available, retrieve such values automatically and present them as choices to the user.

### Environment-specific configuration file

Because available tools differ per machine, declare environment-specific configuration in `.resource-providers.yml`.

- **Location**: `.resource-providers.yml` at the workspace repository root (gitignored)
- **Creation**: Generated by interviewing the user during initial setup
- **Reference**: `/create-recipe` Step 4 and `/design` Phase 3 read this file

If the file is absent, skip automatic resource retrieval and use the conventional interview flow.

### Access method priority

1. **MCP server** — MCP tools available in the session
2. **CLI tool** — Command-line tools on PATH
3. **WebFetch** — Direct access to REST API / OpenAPI specs
4. **Interview** — Fallback when none of the above is available

---

## Format of `.resource-providers.yml`

```yaml
# Environment-specific resource provider configuration
# This file is gitignored. Create one per machine.

providers:
  jira:
    method: mcp
    tool_prefix: "mcp__jira__"           # Prefix to search with ToolSearch
    notes: "Jira Cloud MCP server"       # Note (optional)

  slack:
    method: mcp
    tool_prefix: "mcp__slack__"

  github:
    method: cli
    command: "gh"                         # Command name to detect via which
    auth_check: "gh auth status"         # Auth check command (optional)

  salesforce:
    method: mcp
    tool_prefix: "mcp__salesforce__"

  google_sheets:
    method: mcp
    tool_prefix: "mcp__google_drive__"   # When the Google Drive MCP can also operate on Sheets

  workato:
    method: cli
    command: "workato"
```

### Field descriptions

| Field | Required | Description |
|---|---|---|
| `method` | Yes | One of `mcp`, `cli`, `webfetch` |
| `tool_prefix` | When mcp | Prefix to search with `ToolSearch` (e.g., `mcp__jira__`) |
| `command` | When cli | Command name to detect via `which` (e.g., `gh`) |
| `auth_check` | No | Command that verifies auth state (e.g., `gh auth status`) |
| `base_url` | No | API base URL when using WebFetch |
| `notes` | No | Note (for humans) |

Providers the user does not use need not be listed. Automatic retrieval is attempted only for providers that are listed.

## Per-provider: information to retrieve

For each provider, define "what to retrieve". Concrete tool names and commands follow the configuration in `.resource-providers.yml` and the actual tool schema.

### Jira

| Information to retrieve | Use in the recipe | Hint for retrieval |
|---|---|---|
| Project list (key, name) | Specifying the project key | "list projects"-style tool / API |
| Issue Type list | Specifying the issue type | Retrieved per project |
| Custom fields (name, ID) | Identifying `customfield_XXXXX` | "get fields"-style tool / API |
| Status list | Specifying transitions | Per project / workflow |
| Priority list | Specifying priority | Global setting |

### Slack

| Information to retrieve | Use in the recipe | Hint for retrieval |
|---|---|---|
| Channel list (name, ID) | `channel_id` for message posting | "list channels"-style. Retrieve both name and ID |
| User list (name, ID) | `user_id` for DM targets | "list users"-style |

### GitHub

| Information to retrieve | Use in the recipe | CLI command |
|---|---|---|
| Repository list | Target of triggers / actions | `gh repo list <owner> --json name,description --limit 50` |
| Label list | Labels when creating an issue | `gh label list -R <owner/repo> --json name` |
| Milestone list | Issue milestones | `gh api repos/<owner>/<repo>/milestones --jq '.[].title'` |
| Branch list | Specifying a branch | `gh api repos/<owner>/<repo>/branches --jq '.[].name'` |

### Salesforce

| Information to retrieve | Use in the recipe | Hint for retrieval |
|---|---|---|
| Object list | Specifying the target object | "list objects" / `sf sobject list` |
| Field list (name, type) | Field mapping | "describe object" / `sf sobject describe -s <obj>` |
| PickList values | Pre-resolving choices | `picklistValues` from describe result |
| Record types | Specifying RecordTypeId | `recordTypeInfos` from describe result |

### Google Sheets

| Information to retrieve | Use in the recipe | Hint for retrieval |
|---|---|---|
| Spreadsheet list | Spreadsheet ID | Google Drive search / listing |
| Sheet name list | Specifying the sheet name | Retrieved per spreadsheet ID |
| Header row | Mapping column names | Read row 1 |

### Google Drive

| Information to retrieve | Use in the recipe | Hint for retrieval |
|---|---|---|
| Folder list | Upload destination folder ID | "list files" filtered to folder type |
| File list | Target file ID | "list files" / "search files" |

### Workato

| Information to retrieve | Use in the recipe | CLI command |
|---|---|---|
| Connection list | Connection ID | `workato exec connections.list` |
| Recipe list | Target of `call_recipe` | `workato exec recipes.list` |
| Folder structure | Specifying `zip_name` / `folder` | `workato exec folders.list` |

### Arbitrary REST API

| Information to retrieve | Use in the recipe | Hint for retrieval |
|---|---|---|
| Endpoint list | URL for HTTP actions | Fetch the OpenAPI spec with WebFetch |
| Request / response schema | Defining input / output fields | `schemas` in the OpenAPI spec |
| Authentication method | Reference for connection setup | `securitySchemes` in the OpenAPI spec |

## Procedure inside a skill

### 1. Load environment configuration

Read `.resource-providers.yml`. If the file does not exist, skip automatic resource retrieval.

### 2. Match against target providers

Verify that the provider used in the recipe is defined in `.resource-providers.yml`.

### 3. Detect and run the tool

```
If method is mcp:
  → Search the tool_prefix with ToolSearch
  → Read the schema of the discovered tool and run the appropriate one

If method is cli:
  → Verify existence with which <command>
  → If auth_check is set, verify auth state
  → Run the command from the table above

If method is webfetch:
  → Access the API using base_url
```

### 4. Present to the user

Present the retrieved information as choices to the user:

```
I have retrieved the resource information. Please choose from the following:

**Jira projects:**
1. DEV - Development
2. OPS - Operations
3. IT - IT Support

**Issue Type (DEV):**
- Bug, Story, Task, Epic

Which project should the issue be filed under?
```

### 5. Fallback

In the following cases, silently fall back to the conventional interview:
- `.resource-providers.yml` does not exist
- The target provider is not defined
- Tool detection failed
- An error occurred during resource retrieval

Do not display an error message; transition naturally into the interview flow.

## Initial setup

When `/create-recipe` or `/design` is run without `.resource-providers.yml`, guide setup as follows:

```
The configuration file for automatic resource retrieval (.resource-providers.yml) does not exist yet.
Would you like to configure it so that resource information for external services
(Jira project list, Slack channels, etc.) can be retrieved automatically?

If yes, tell me which MCP servers and CLI tools you use:
- Do you use a Jira MCP? → What is the tool name prefix? (e.g., mcp__jira__)
- Do you use a Slack MCP?
- Is the gh CLI installed?
- Any other MCP servers or CLIs available?

To skip, type "skip" (we will proceed with the conventional interview instead).
```
