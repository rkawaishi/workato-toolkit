## 1. アクターと前提

### 1.1 アクター

| アクター | 役割 | 主に使うもの |
|---|---|---|
| **管理者** | ワークスペース初期化、API キー払出し、昇格ポリシー決定、prod 承認 | `/setup-workspace` `/issue-api-keys`、Workato UI（承認・ロール作成） |
| **開発者** | 仕様策定〜ビルド〜dev での push・テスト・修正、Policy A なら dev→test 昇格 | `/spec`〜`/implement`、`/workato-create`、`/push-project`、`/deploy-project`（dev→test）、（新）`/run-recipes` `/diagnose-jobs` |
| **運用担当** | 昇格後の稼働確認、障害の一次対応、回復の確認 | （新）`/inspect-env`、`/deploy-project`、Workato UI（prod の停止・ジョブ再実行）。小さい org では開発者と同一人物（dev 側の修正はそのとき開発者として `/diagnose-jobs` で行う） |
| **エージェント**（Claude Code + 本プラグイン） | 上記アクターの作業を代行・支援する。**環境ごとに能力が非対称**（下表） | — |

### 1.2 環境と権限の前提（前 spec §5 の帰結。本書の全ストーリーが従う）

| | dev | test（=Staging） | prod |
|---|---|---|---|
| エージェントの書込（push / recipes start・stop / sdk push、テーブル行 CRUD 等のヘルパー新設コマンド） | ✅ | ❌（読取専用キー + hook #10 が物理ブロック。**hook が握るのは push/start・stop/sdk push のみ — テーブル行 CRUD 等の新設コマンドはヘルパー内蔵の dev ガード + 権限マトリクス改修（§4）で塞ぐ**） | ❌（同左） |
| エージェントの読取（jobs / recipes / pull / deploy status） | ✅ | ✅（読取専用キーで可、実機確認は §6） | ✅（同左） |
| 変更の反映経路 | push | Deploy（dev→test。Policy A ならエージェント実行可） | Deploy（test→prod。実行前にユーザ明示確認、**承認は常に UI の人間**） |
| レシピの起動・停止 | エージェント可 | 人間が UI で | 人間が UI で |

この非対称が運用ストーリーの骨格になる: **dev では「エージェントが直す」、test/prod では
「エージェントは見る・原因を特定する、直すのは dev 経由の再昇格、触るのは人間」**。

### 1.3 障害の検知経路（前提の明確化）

本プラグインは**オンデマンド実行**であり、常駐監視はスコープ外（§4）。したがって障害の入口は:

1. **push 直後**: `/push-project --test` → `/diagnose-jobs` の検証ループ（開発者がその場にいる）
2. **昇格直後**: `/deploy-project` 完了後に `/inspect-env <env>` で稼働確認（運用担当がその場にいる）
3. **定常運用中**: Workato 側のアラート（レシピ停止・ジョブ失敗のメール等）や業務側の気づきを
   **人間が受け取り**、セッションを開いて `/inspect-env` / `/diagnose-jobs` を叩く

(3) を自動化したければ Workato のアラート設定・外部監視の領分。`/inspect-env` は
「人間が定期的に叩く」運用には使える（cron 的な常駐はしない）。

## 2. 通しウォークスルー（一次資料 — ストーリーはここから導出する）

題材: **「Salesforce の商談が Closed Won になったら、会計 SaaS に請求書ドラフトを作成し
Slack に通知する」レシピ 1 本**。トリガーはポーリング型（テスト実行が最も厄介なタイプ）、
コネクション 3 本（Salesforce / 会計 SaaS / Slack）。org は Dev / Staging(TEST) / Prod の
3 環境、昇格ポリシー A（dev→test はエージェント実行可）。

各ステップの右に、担保するストーリー ID（§3）を付す。**太字**は本 spec で新設・変更される箇所。

### Phase 0 — 準備（管理者、初回のみ）

1. `/setup-workspace` — リポジトリ骨格、ヘルパー実体化、秘匿 deny-list、プロファイル登録
2. `/issue-api-keys` — dev=書込可 / test・prod=読取専用の 3 キーを払出し、keyring 登録。
   test/prod プロファイルで push が拒否されることをスモーク確認
3. （任意）`/onboard` — 既存資産からナレッジベース初期化

### Phase 1 — 仕様策定〜ビルド（開発者）

4. `/spec` → `/plan` → `/tasks` → `/implement`（小さければ直接 `/workato-create recipe`）。
   産物: `projects/billing-sync/` にレシピ JSON + コネクション JSON 3 本。
   spec には**期待入出力**（商談レコード → 請求書ドラフトのフィールド対応表）が書かれている
   — Phase 2 のテスト照合（S5-2）はこれを期待値として使う。
   **設計規約（S8-5）**: 会計 SaaS のエンドポイント URL・テナント ID など**環境で変わる値は
   レシピにハードコードせず Environment properties を参照**する（dev の値はエージェントが
   `workato properties set` で設定できる。これを守らないと Phase 3 以降、環境切替のたびに
   レシピ本体の書き換えが必要になる）

### Phase 2 — dev へ push してテスト（開発者）【本 spec の中心 1/3】

5. `/push-project --test` — 環境ガード（プロファイル `-dev` 確認）→ 検証 → **新規コネクション
   3 本を先に push** → ユーザに UI での認証を依頼（エージェントには資格情報が無い。人間の作業）
   → 残り資産を push → レシピ起動。
   分岐: **レシピが起動に失敗する（S6-4）** — push 成功・validate 通過でも、壊れた/不完全な
   レシピ定義・connection 未割当・トリガー設定不備は**起動時に初めてエラーになる**。起動に
   失敗したレシピは**ジョブが 1 件も流れない**ため、手順 8 のジョブ診断では捕捉できない —
   起動エラー本文を入口とする S6-4 の修正ループ（修正 → 再 push → 再起動）を先に回し、
   起動が通ってから手順 6 へ進む
6. **テストジョブを能動的に発生させる（S5-1）**: トリガーがポーリング型なので、push しただけでは
   ジョブは流れない。エージェントは
   a. テストデータの準備を依頼する — 「Salesforce sandbox で商談を 1 件 Closed Won にしてください」
      （**S5-3**: 外部システム側の操作はエージェントの手が届かない境界。手順書を出して人間に渡す）
   b. ポーリング間隔ぶん待つ（poll-now 相当の API が有れば即時発火 — §6 OQ）
   c. webhook トリガーであれば curl POST、Workflow App であればフォーム送信依頼、と
      **トリガー型ごとに投入手段が変わる**（§3 S5-1 のマトリクス）。
   分岐: レシピが **Data Table / Lookup Table に依存**する場合（S5-4）— テスト前に
   テーブルへテストレコードをシードする（Data Table 行トリガーならこのレコード作成が投入
   手段そのもの）。**掃除は修正ループの終了時**（手順 8 のループ中は周回ごとにレコードが
   増える — ID を記録し、手順 9 の commit 前にまとめて消す）
7. **結果を期待値と照合する（S5-2）**: `jobs list` でジョブ検出 → `jobs get` で入出力を取得 →
   Phase 1 の spec のフィールド対応表と突き合わせ。「ジョブが成功した」だけでは合格にしない
   （成功して間違ったデータを作るのが一番怖い）
8. 失敗した場合 → **`/diagnose-jobs`（S7-1, S7-2）**: エラー本文から原因分類
   （datapill 参照 / フィールド不一致 → エージェントが直せる。connection 未認証 / 外部 API 側 →
   人間に引き渡す）→ 直せるものは**既定で 修正 → 再 push（--restart-recipes）→ 再検証 を
   green までループ**（dev 限定。ユーザ確認済み 2026-07-07）。
   分岐: **エージェントがレシピを書ききれない（S7-4）** — ループが上限に達した、または
   LLM が生成できない箇所（複雑な formula・UI でしか設定できない項目）と判明した場合、
   **人間に UI での修正を依頼**する。修正完了の合図を受けて `/pull-project` で取り込み →
   ローカルとの diff を提示 → git commit → 差分から `/learn-recipe` で知見化（次回の生成
   精度に還元）。**取り込み前の再 push は人の修正を上書きして消すため厳禁** — UI 引き渡し
   以降、pull が完了するまで push を封じる
9. green になったら **`/run-recipes` で稼働一覧を確認し（S6-2）、検証専用に動かしたレシピは
   停止する（S6-1）** → git commit + push（`/push-project` は git に触れない — 既存規約）

### Phase 3 — test へ昇格して検証（開発者→運用担当）

10. `/deploy-project preview` → `run --to test` → 昇格チェックリスト
    （test 側コネクション認証・environment properties・**test でのレシピ起動 — いずれも
    人間の UI 作業**。Deploy が稼働状態を引き継ぐかは §6 OQ）→ `status --wait`。
    **properties は「seeded ですか」の一言でなく、エージェントが dev の properties 一覧から
    「test で設定すべき名前 + dev での値 + 記入欄」の設定リストを生成して人間に渡す（S8-5 —
    test/prod への upsert はエージェントの権限外）**。**Lookup / Data Table も同様に、
    テーブル名・要件つきのシードリストを生成して人間へ（S8-6 — Deploy はコードを運ぶが
    テーブルの中身は運ばない）**。
    分岐: deploy が failed で終端 → マニフェスト内容とエラーを報告し、dev へ差し戻す
11. **`/inspect-env test`（S8-1）**: レシピが test で稼働しているか、ジョブが流れて成功して
    いるかを読取専用で確認。ここで test 用のテストデータ投入が再度必要なら手順 6 を test の
    外部システム（sandbox）に対して繰り返す。
    分岐: コネクション未認証で全ジョブ失敗 → チェックリスト消化漏れとして人間へ差し戻し

### Phase 4 — prod へ昇格（運用担当 + 管理者）

12. `/deploy-project run --to prod` — ユーザの明示確認 + **承認は管理者が UI で**（自動化恒久禁止）。
    prod でのレシピ起動も人間の UI 作業。
    分岐: 承認が却下された → 理由を確認し、dev での修正（Phase 2）へ差し戻す
13. `/inspect-env prod`（S8-1）: 初ジョブが流れて成功するまで確認。以後、定常運用へ

### Phase 5 — 定常運用: 本番障害（運用担当）【本 spec の中心 2/3】

14. 数週間後、会計 SaaS の API 仕様変更で prod のジョブが連続失敗。**検知は Workato アラート
    メール**（§1.3 (3)）— 運用担当がセッションを開く
15. **`/inspect-env prod`（S8-2）**: 読取専用で失敗ジョブのエラー本文・失敗率・影響レシピを
    特定し、prod のレシピ定義を scratch へ pull して dev と diff。可能なら**変更履歴
    （activity log — §6 OQ）で「誰かが UI で直近に触っていないか」も照合**。
    → 「定義は dev と同一。外部 API のレスポンス形式変更が原因」＝**外部仕様変更
    （レシピ側の適応修正が必要）**と分類（一時的なレート制限・先方障害なら引き渡し/回復待ちで、
    以降の手順は不要）
16. **緊急止血の判断（S6-3）**: 失敗ジョブが積み上がるだけなら止めない選択もある。止める
    場合、**エージェントは prod のレシピを止められない**（1.2 の非対称）— レシピ名・ID と
    UI の該当 URL を提示して人間に停止してもらう
17. **hotfix 経路（S8-4）**: dev で修正 → Phase 2 の手順 6〜8 で再現・修正・検証（
    `/diagnose-jobs` のループ）→ `/deploy-project run --to test` → `/inspect-env test` →
    `run --to prod`（確認+承認）→ `/inspect-env prod` で回復確認。
    **prod を直接直す近道は存在しない**（キーと hook が物理的に塞いでいる）
18. **取りこぼしの回収（S7-3）**: 障害期間中に失敗したジョブの再実行。rerun API が有るかは
    §6 OQ — 無ければ「UI のジョブ再実行手順 + 対象ジョブ一覧の提示」まで がエージェントの仕事
19. 復旧後、`/learn-recipe` / org/docs へ「この connector はレスポンス形式が変わりうる」等の
    知見を書く — 学習ループが閉じる

### Phase 6 — 昇格自体が悪かった場合（S8-3、Phase 5 の変形）

20. 新バージョンを昇格した直後に業務エラー多発 → 原因は新バージョン自体 → **ロールバック**:
    Deploy API に「過去 deployment の再適用」が有るかは §6 OQ。無い場合の正規手順は
    **git 上の直前リリース状態を dev に復元 → 通常経路で再昇格**（git がリリース台帳になる。
    Phase 2 の手順 9 で commit している事がここで効く）

## 3. ストーリー詳細（受け入れ基準つき）

記法: 各ストーリーは「アクター / 前提 / トリガー / 主フロー / 受け入れ基準（AC）/
失敗・分岐 / 現状とギャップ」。現状表記: ✅=充足 / △=手段はあるがワークフロー未整備 / ❌=手段なし。

### S5-1 テストジョブを能動的に発生させたい

- **アクター**: 開発者（ウォークスルー手順 6）
- **前提**: `/push-project` で dev に push 済み、レシピ起動済み、コネクション認証済み
- **トリガー**: `--test` 指定、または push 完了後にテストへ進むと決めた時
- **主フロー**: トリガー型を recipe JSON の trigger adapter から判定し、型別に投入する:

| トリガー型 | 投入手段 | 人間の作業か |
|---|---|---|
| webhook（realtime） | webhook URL へ curl POST（URL の機械的取得可否は §6 OQ） | エージェント可 |
| ポーリング | ソース側にテストデータ作成を依頼 → 次回ポーリング待ち（poll-now の有無は §6 OQ） | **依頼（S5-3）** |
| スケジュール | 次回実行を待つ／一時的に間隔を縮めて push（戻し忘れ防止の注意書き必須。**dev 限定の手段** — test/prod では待つのみ） | エージェント可 |
| Workflow App フォーム | フォーム送信を依頼（URL 提示） | **依頼** |
| MCP サーバ / Genie | AI クライアントからの呼出しを依頼（MCP サーバは push-project §4 の既存チェックリスト。Genie 用チェックリストは未整備 — §4 の /push-project 改修で新設） | **依頼** |
| Data Table 行トリガー | Data Table にテストレコードを作成/更新して発火（S5-4。New/updated 系トリガーは既存行の**更新**でも発火させる） | エージェント可／rows API 無しなら依頼（§6 OQ） |
| API エンドポイント（API Platform） | エンドポイント URL へ curl（クライアントトークン要 — API Platform 側の設定は別系統。/issue-api-keys の混同ガード参照） | エージェント可（トークンは人間が用意） |
| その他（Event Streams 消費・Recipe Function 呼出し等） | 上流（publish 側レシピ / 呼出し元レシピ）からの発火。発火手段の詳細は実機で確認（§6 OQ） | 型による |

- **AC**:
  - [ ] マトリクスの**全行**に対し「何をすればジョブが 1 件流れるか」が手順化されている
        （行が増えたら AC も追従する — 固定数を書かない）
  - [ ] 人間に依頼する場合、コピペで実行できる具体的指示（URL・レコード条件つき）を出す
  - [ ] 投入後、ジョブ出現まで待機し S5-2 の照合へ自動で接続する
- **失敗・分岐**: ジョブが一定時間出現しない → レシピ停止中/トリガー条件不一致/ポーリング
  未到達を切り分けて報告（無限待ちしない）
- **現状**: ❌ `--test` は「失敗ジョブの有無を見る」だけ。投入手段が未整理

### S5-2 テスト結果を期待値と照合したい

- **アクター**: 開発者
- **前提**: S5-1 でジョブが 1 件以上流れた。spec（または依頼内容）に期待入出力がある
- **主フロー**: `jobs get` で入出力を取得 → 期待値（spec のフィールド対応表）と突合 →
  相違点を「ジョブは成功したが値が違う」を含めて報告
- **AC**:
  - [ ] 「ジョブ成功 = テスト合格」としない。出力フィールドの値まで照合する
  - [ ] 期待値が無い場合、照合前にユーザへ期待値を確認する（勝手に合格させない）
  - [ ] 不一致は S7-1 の原因分類に接続する
- **失敗・分岐**: 出力が暗号化/マスクされ取得不能 → その旨と UI での確認手順を提示
- **現状**: △ `jobs get` はあるが照合ワークフローなし

### S5-3 テストデータの準備は外部システム側の作業（境界の明示）【新設】

- **アクター**: 開発者（依頼を受けるのは人間、またはソース側 SaaS の別ツール）
- **内容**: ソースシステム（例: Salesforce sandbox）へのテストレコード作成は Workato の
  API では代行できない。エージェントは**作成すべきレコードの条件（トリガー条件を満たす
  フィールド値）を具体化して依頼する**ところまでが仕事
- **AC**: [ ] 依頼文にトリガー条件を満たす具体値が含まれる / [ ] 作成後の合図で S5-1 の待機を再開する

### S5-4 Data Table / Lookup Table 依存レシピのテストデータを登録・編集・削除したい（dev）【新設】

- **アクター**: 開発者
- **前提**: レシピが Data Table（トリガー 4 種・アクション 10 種）または Lookup Table に
  依存している。**依存はアダプタ（トリガー/アクション）だけでなく formula 参照でも生じる**
  （Lookup Table の `lookup()`、Data Table の formula lookup — kit doc）。S5-3 と違い
  **テーブルは Workato 内のリソース** — 外部 SaaS ではないので、API が届くなら
  エージェントが自分で用意できる
- **トリガー**: S5-1 のテスト実行の前後（シード → 発火 → 検証 → 掃除）
- **主フロー**:
  1. **依存特定**: recipe JSON からテーブル参照を洗い出す — アダプタのステップに加え、
     **formula モード内のテーブル参照も走査**する
  2. **シード**: テストレコードを登録（Lookup Table は行 CRUD の Developer API が公開されて
     いる — ヘルパー拡張候補。Data Table の rows API の有無・パスは §6 OQ。API が無い型は
     UI での登録を依頼）
  3. **発火**: Data Table 行トリガーのレシピは、レコードの**作成**が投入手段そのもの。
     **New/updated 系トリガーは既存レコードの「更新」でも発火を確認**する（作成だけでは
     トリガー 4 種のうち半分しか検証できない）
  4. **検証**: レシピがテーブルへ書き込む側なら、実行後にレコードを読み出して S5-2 の照合に含める
  5. **掃除**: テストで作った/更新したレコードを片付け、dev のテーブルをテスト前の状態に戻す。
     **掃除のタイミングはループ終了時**（S7-2/S6-4 の修正ループ中は各周回でレコードが増える —
     作成したレコード ID を周回をまたいで記録し、green 到達または引き渡しの時点でまとめて消す。
     手順 9 の git commit 前に完了させる）
- **AC**:
  - [ ] シード・**更新**・削除がテスト手順の一部として明記される（「テーブルは既にある」前提にしない）
  - [ ] 掃除まで含めて「テスト完了」とする（作ったレコードの ID を**ループ横断で**控えて消す）
  - [ ] テーブル書込は **dev 限定**（ヘルパー拡張に dev ガード内蔵。test/prod キーの権限で
        テーブル書込が塞がれていることは §4 の /issue-api-keys マトリクス改修で担保 —
        現行マトリクスには Lookup Tables の行が無い）
  - [ ] `Truncate table` 系の全削除は使わない（既存データを巻き込む — 自分が作ったレコードだけ消す）
- **失敗・分岐**: テーブルのスキーマがレシピの期待と不一致（列 ID 変更等）→ S7-1 の
  フィールド不一致として診断へ / rows API が無い → 登録・削除の手順書を出して人間に依頼 /
  S7-4 で UI 修正へ引き渡す場合 → **作成済みテストレコードの一覧を引き渡し文に含める**
  （人間のテスト中に消さない・人間のテスト後に掃除する）
- **現状**: ❌ ストーリー自体が無かった。昇格チェックリストの「Lookup・Data Table シード」
  1 行（S8-6 が引き取る）と /issue-api-keys の権限行（Data Tables 作成/変更 = dev のみ）が
  あるだけで、テスト実行との接続が無い（ユーザレビュー指摘 2026-07-07）

### S8-4 本番障害の修正は dev 経由で最短昇格したい（hotfix 経路）【新設】

- **アクター**: 開発者 + 運用担当
- **前提**: S8-2 で「定義差・レシピ欠陥」または「外部仕様変更（適応修正が必要）」と分類された
- **主フロー**: ウォークスルー手順 17。dev で再現（可能なら）→ `/diagnose-jobs` ループ →

- **AC**:
  - [ ] 「prod を直接直す」選択肢が一切提示されない
  - [ ] 通常昇格と同じガード・チェックリストを通る（緊急時ほど省略しがちな箇所を checklist が支える）
  - [ ] 回復確認（S8-1）と取りこぼし回収（S7-3）まで含めて「完了」とする
- **現状**: △ 各部品はあるが、経路として文書化されていない

### S8-5 環境ごとの接続先・設定値を properties で切り替えたい【新設】

- **アクター**: 開発者（設計・dev 値の設定）+ 運用担当/管理者（test/prod 値の設定は人間）
- **前提**: 接続先エンドポイント URL・テナント ID・通知先など、**環境で値が変わる設定**が
  レシピにある（kit doc `platform/environment-properties.md` の用途例そのもの）
- **トリガー**: ビルド時（設計規約として）と昇格時（target env の値の用意）
- **主フロー**:
  1. **ビルド時（規約）**: 環境依存値はハードコードせず Environment properties 参照で生成する
     （`/workato-create` の生成リファレンスに規約として明記）。dev の値はエージェントが
     `workato properties set` で設定する
  2. **昇格時**: dev の `workato properties list` から、レシピが参照している名前を突き合わせ、
     **target env 用の設定リスト**（名前 / dev の値 / target の値の記入欄）を生成して人間へ。
     test/prod への upsert は読取専用キーの権限外（前 spec §5 マトリクス: upsert は dev のみ ✅）
     — 設定は人間が UI（Tools > Environment properties）で行う
  3. **検証**: `/inspect-env` が「properties 未設定/名前欠落」を環境差として検知する
     （S8-2 の環境差分類の具体化。読取専用キーで properties list が通るかは §6 OQ）
  4. **注意の伝達**: properties の値は**ジョブ開始時に凍結**され実行中ジョブに反映されない
     （kit doc）— 値を変えたら該当レシピの再起動が必要（dev は `/run-recipes restart`、
     test/prod は人間の UI 操作）
- **AC**:
  - [ ] 生成されるレシピに環境依存値のハードコードがない（properties 参照になっている）
  - [ ] 昇格チェックリストの properties 項目が「seeded ですか」の一言でなく、**名前つきの
        設定リスト**として出力される
  - [ ] test/prod の値設定を人間の作業として明示し、エージェントは upsert を試みない
  - [ ] 値変更後の再起動要否が案内される
- **失敗・分岐**: target env で properties 未設定のままレシピを起動 → 誤った接続先に接続 or
  ジョブ失敗 → S8-1 の稼働確認が「環境差」として検知し、設定リストへ差し戻す
- **現状**: △ kit doc と deploy チェックリストの 1 行（"Environment properties are seeded"）
  のみ。設計規約（ハードコード禁止）・設定リスト生成・切替検証のワークフローが未整備
  （ユーザレビュー指摘 2026-07-07）。前 spec の「/properties スキルは CLI で充足のため
  見送り」は維持 — 独立スキルにせず、`/workato-create`（規約）・`/deploy-project`（リスト
  生成）・`/inspect-env`（検証）に織り込む

### S8-6 昇格時に Lookup / Data Table を target 環境へシードしたい【新設】

- **アクター**: 開発者（シード内容の設計）+ 運用担当/管理者（test/prod への投入は人間）
- **前提**: Deploy のマニフェストは**コードを運ぶがデータは運ばない** — テーブルの定義は
  昇格されても、中身（マスタデータ・参照データ）は空のまま。properties（S8-5）の
  テーブル版で、レシピは「テーブルに然るべき行がある」前提で動く
- **トリガー**: `/deploy-project` の昇格チェックリスト消化時
- **主フロー**:
  1. レシピが参照するテーブルと、その**シード要件**（マスタ系=全行必要 / 蓄積系=空で開始 /
     参照系=環境ごとに値が違う行）を分類してリスト化
  2. dev の該当テーブルから**シードリスト**（テーブル名 / 列構成 / 持っていくべき行 or
     環境別に値を変える行）を生成して人間へ — test/prod への書込はエージェントの権限外
  3. 投入後、`/inspect-env` が「テーブル存在・行数」を読取確認（読取可否は §6 OQ）
- **AC**:
  - [ ] チェックリストが「シードしましたか」の一言でなく、**テーブル名と要件つきのリスト**になる
  - [ ] マスタ系・蓄積系・環境依存系の区別がリストに現れる（蓄積系まで丸ごとコピーさせない）
  - [ ] test/prod への投入は人間の作業として明示される
- **失敗・分岐**: シード漏れのままレシピ起動 → lookup 失敗・トリガー無発火として顕在化 →
  S8-1 の稼働確認が検知し、シードリストへ差し戻す
- **現状**: △ deploy チェックリストに「Lookup・Data Table シード」の 1 行のみ。
  何をどう持っていくかの手順・リスト生成が無い（ユーザレビュー指摘 2026-07-07）

## 5. 決定事項と前提（レビューで覆す前提で明文化）

1. **自動修正の範囲（ユーザ確認済み 2026-07-07）**: `/diagnose-jobs` は**既定で自動修正と
   再 push まで実行**する（dev 限定）。直せない分類（connection 認証・権限不足、
   外部の一時障害 — S7-1 の表）はユーザーへ引き渡す。診断のみは `--no-fix`。test/prod への修正は常に deploy 経由
   （hook #10 が物理的に下支え）
2. **読取専用の担保は 3 層**: スキル記述（書込コマンド不使用）+ 権限（test/prod キーは
   読取専用 — /issue-api-keys のマトリクス）+ hook #10 の物理ブロック
3. **境界では「案内」まで が仕事**: エージェントが代行できない操作（UI 認証、フォーム送信、
   prod の停止、テストデータ作成、prod 承認、UI でのレシピ修正）は、**具体的な対象・URL・
   手順を揃えて人間に渡す**ところまでをスキルの責務とする（S5-3 / S6-3 / S7-4 / S8-3 の AC）
4. **UI 修正の取り込み規律（S7-4）**: 人間が UI で修正した瞬間、リモートが正・ローカルは
   stale になる。**引き渡しから `/pull-project` 完了まで push を封じ**、取り込み後は
   diff → git commit → `/learn-recipe` で必ず知見化する（LLM が書ききれなかった箇所こそ
   組織知識の最優先候補 — 取り込んで終わりにしない）
5. **「push 成功」「deploy 成功」「ジョブ成功」を合格にしない**: push 成功・validate 通過でも
   起動で落ちる（S6-4）、deploy 成功でも業務は動いていない（S8-1）、ジョブ成功でも値が違う
   （S5-2）— 各段階の「本当の合格条件」はそれぞれのストーリーが定義する
6. スキル名は動詞-名詞規約: `/run-recipes` `/diagnose-jobs` `/inspect-env`（#16 の接頭辞判断
   が出たら追従）
7. S5-1 のテスト投入は**独立スキルにしない**（push 直後の文脈に強く従属するため
   `/push-project --test` の一部。`/diagnose-jobs` の再投入からも同じマトリクスを参照する）
